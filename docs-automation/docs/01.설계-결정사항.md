## 2. 확정된 아키텍처 개념

### 2.1 전체 파이프라인 흐름

```
+---------------------------------------------------------+
|               각 제품 레포 - 분산 처리                     |
|                                                          |
|  main branch -- git push --> CI 트리거                    |
|                                |                          |
|                         [코드 분석 + AI .md 생성]          |
|                                |                          |
|                         docs-auto 브랜치 <- .md Push       |
|                         (main 코드와 분리)                 |
+---------------------------------------------------------+
                         |
                         v
+---------------------------------------------------------+
|              중앙 처리 - 스케줄 기반 (배치)                 |
|                                                          |
|  특정 시간에 각 레포의 docs-auto 브랜치 Pull                |
|       |                                                  |
|  YAML frontmatter 기반 docs/ 폴더 분류                    |
|       |                                                  |
|  sidebars.js 자동 갱신 -> Docusaurus 빌드 -> 배포          |
+---------------------------------------------------------+
```

### 2.2 아키텍처 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| 파이프라인 트리거 | Git push | 개발자 추가 작업 불필요 |
| AI 개입 시점 | 처음부터 전면 개입 | 코드에 주석이 거의 없어 파싱만으로는 유용한 문서 생성 불가 |
| 레포 구조 | 제품별 개별 레포 | 기존 인프라 유지 |
| .md 저장 위치 | 각 레포의 docs-auto 브랜치 | main 코드 오염 방지 |
| 중앙 처리 방식 | 스케줄 기반 배치 | 매 push마다 빌드하지 않고 일괄 처리 -> 빌드 비용 절감 |
| 문서 구조 | 페이지 단위 테마 (실제 사이트 1:1 매핑) | 실제 Docusaurus 사이트 구조 기반 |
| 문서 업데이트 방식 | 모듈 단위 전체 재생성 + 프로젝트 단위 증분 | 문서 내 일관성 보장 + 불필요한 재생성 방지 |
| 콜드 스타트 | 점진적 (구조 뼈대 -> push마다 채움) | 한 번에 전체 생성은 볼륨 과대 |
| 오케스트레이션 구조 | **1단계** (Main이 직접 오케스트레이션) | 서브 에이전트 중첩 제약 해소 (v4), Main이 판단+오케스트레이션 직접 수행 |
| Full Reset | 매 테마마다 docu-writer/critic 신규 생성 | 컨텍스트 오염 방지, 비용 효율(1.0x vs Resume 1.8~3.6x) |
| 메인 역할 범위 | 루프 관리 + 판단 + 요구사항서 + Agent 위임 + 재시도 + 저장 | v4에서 task-orchestrator 흡수 |
| 반환 형식 | critic 4필드(result/frontmatter_valid/theme_fitness/critic_feedback) | critic -> Main 간 구조화된 I/O 계약 |

### 2.3 v3 -> v4 변경: 오케스트레이션 평탄화

**변경 이유**: Claude Code에서 서브 에이전트가 서브 에이전트를 생성할 수 없음.

v3 구조 (3단계 중첩):
```
Main -> task-pipeline(Agent) -> task-orchestrator(Agent) -> docu-writer/critic(Agent)
```

v4 구조 (1단계 평탄):
```
Main CLI (Level 0): 테마 루프 + 판단 + 요구사항서 + 오케스트레이션 + 재시도 + 저장
  |-> docu-writer (Level 1 Agent): md 작성 (리프)
  |-> critic (Level 1 Agent): 독립 검증 (리프)
```

| 비교 항목 | v3 (2단계 오케스트레이션) | v4 (1단계 오케스트레이션) |
|-----------|------------------------|------------------------|
| Agent 중첩 | 3단계 (불가능) | 1단계 (실행 가능) |
| 에이전트 수 | 4개 (pipeline, orchestrator, writer, critic) | 2개 (writer, critic) |
| 스킬 수 | 5개 | 4개 |
| Main 역할 | 루프 관리 + 상태 추적만 | 루프 + 판단 + 요구사항서 + 재시도 + 저장 |
| 복잡도 | 에이전트 간 I/O 계약 7개 | I/O 계약 5개 |
| Full Reset | 동일 (매 테마마다 신규 생성) | 동일 |
| 품질 보증 | 동일 (docu-writer + critic 역할 분리) | 동일 |

**결론**: 실행 불가능한 구조를 실행 가능하게 변경하면서, 역할 분리(md 작성/검증)와 Full Reset 전략은 유지.

## 3. 핵심 설계: 순차적 관점 전환 방식

### 3.1 기본 원리

코드 변경에 대해 테마를 순차 순회하면서, 각 테마 관점에서 4단계 사이클을 반복한다:

| 단계 | 수행 | 내용 |
|------|------|------|
| **판단** | Main | git diff와 테마 정의를 분석하여 "이 변경이 해당 테마 문서에 반영이 필요한가?" 판단. Yes이면 요구사항서 작성 |
| **작성** | Main -> Agent(docu-writer) | 요구사항서 + 관련 코드를 docu-writer에 전달하여 .md 작성 |
| **검증** | Main -> Agent(critic) | 작성된 .md를 critic에 전달하여 frontmatter 유효성 + 테마 적합성 검증. Fail 시 최대 2회 재작성 |
| **저장** | Main | md_content를 .md 파일로 저장 + execution-log 기록 |

> 상세 실행 흐름, 에이전트 구성, 4단계 사이클 구현: [02.아키텍처.md](02.아키텍처.md) 섹션 2~3 참조

### 3.2 이 방식을 선택한 이유

**대안 (병렬 분류 -> 병렬 생성)과의 비교:**

| 비교 항목 | 순차적 관점 전환 (채택) | 병렬 분류 -> 생성 |
|-----------|----------------------|-----------------|
| 누락 위험 | **낮음** - 모든 테마를 순회하므로 빠짐 없음 | 높음 - 분류 단계에서 AI가 테마를 누락할 수 있음 |
| 판단 난이도 | **낮음** - "이 테마에 필요한가?" Yes/No | 높음 - "어떤 테마들에 영향을 주는가?" 다중 분류 |
| 문서 간 일관성 | **유리** - 이전 테마 결과를 다음 테마 생성 시 참조 가능 | 불리 - 독립 생성이라 중복/불일치 가능 |
| LLM 호출 횟수 | 많음 (최대 테마 수 x 4) | 적음 (분류 1회 + 해당 테마 N회) |
| 총 실행 시간 | 길음 (순차) | 짧음 (병렬 가능) |

**결론**: 품질과 안전성 우선. 비용/시간은 최적화로 해결 가능.

## 4. 테마 정의 (페이지 단위)

### 4.1 설계 변경: 제품별 7테마 -> 페이지 단위 테마

**변경 이유**: 실제 Docusaurus 사이트(http://110.110.10.70:8080/) 분석 결과, 초기 추정 기반 7테마 구조와 실제 사이트 구조가 불일치.

| 항목 | 변경 전 (v2) | 변경 후 (v3) |
|------|-------------|-------------|
| 테마 단위 | 제품별 역할 (overview, features 등) | **페이지 단위** (1 테마 = 1 md 파일) |
| 입력 계약 | `theme` + `product_id` + `output_path` | `theme` + `section` + `output_path` |
| 테마 ID 예시 | `overview` | `getting-started/requirements` |
| 확장 방식 | 테마 7개 고정, 제품별 반복 | 테마를 추가하여 페이지 단위로 확장 |

**결정 근거**:
1. 실제 사이트의 "시작하기" 섹션(3페이지)은 제품별이 아닌 시스템 전체 범위 -> 기존 `overview` 테마로 매핑 불가
2. 제품별 서브페이지가 균일하지 않음 (MiRcsServer 6페이지 vs MiRcsClient 3페이지) -> 고정 7테마로 모든 제품 처리 불가
3. 1 테마 = 1 md 파일로 단순화하면 기존 4단계 사이클을 변경 없이 재사용 가능

### 4.2 1차 스코프 (4개 테마)

| # | 테마 ID | 테마명 | 섹션 | 문서의 관점 | 대상 독자 | docs 경로 |
|---|---------|--------|------|-----------|----------|----------|
| 1 | `intro` | 개요 | intro | 프로젝트 전체 소개 | 모든 방문자 | `docs/intro.md` |
| 2 | `getting-started/requirements` | 시스템 요구사항 | getting-started | 설치/실행에 필요한 환경과 조건 | 설치자, 운영자 | `docs/getting-started/requirements.md` |
| 3 | `architecture/overview` | 제품 아키텍처 | architecture | 시스템 전체 구조와 모듈 간 관계 | 신규 팀원, 개발자 | `docs/architecture/overview.md` |
| 4 | `architecture/component-diagram` | 컴포넌트 다이어그램 | architecture | S/W별 대략적인 컴포넌트 구성 | 개발자 | `docs/architecture/component-diagram.md` |

### 4.3 2차 확장 예정

| 항목 | 내용 |
|------|------|
| 시작하기 확장 | 설치 가이드, 빠른 시작 (2페이지 추가) |
| 제품별 페이지 | MiRcsServer 등 6개 제품의 서브페이지 테마 추가 (28페이지) |
| 라이브러리 | Utility/Engine/Infra/3rd Party 라이브러리 페이지 |
| 기타 섹션 | 개발 가이드, CI/CD, API & 프로토콜, 참조 |

## 5. YAML Frontmatter 스키마 (9개 필수 + 2개 선택)

```yaml
---
# 필수 필드
title: "문서 제목"                    # 페이지 제목
sidebar_label: "사이드바 표시 이름"     # 사이드바에 표시될 짧은 이름
sidebar_position: 3                  # 사이드바 내 순서
section: "getting-started"           # 사이트 섹션
theme: "getting-started/requirements" # 테마 ID

# 자동 생성 메타데이터
auto_generated: true                 # 자동 생성 여부
source_files:                        # 원본 소스 파일 목록
  - "CMakeLists.txt"
  - "src/server/main.cpp"
last_commit: "abc1234"               # 마지막 분석한 커밋 해시
generated_at: "2026-03-05T14:30:00"  # 생성 시각

# 선택 필드
auto_generated_warning: ""           # 검증 미통과 시 경고 메시지
tags: []                             # 검색/분류용 태그
---
```

## 6. 콜드 스타트 전략 (점진적)

### Day 0: 구조 뼈대 생성
- 코드 내용을 분석하지 않음
- 디렉터리 트리 + CMakeLists.txt + 파일명만으로 문서 목차 수준의 뼈대 생성
- 예: `overview.md`에 "{제품명}은 [분석 예정] 애플리케이션입니다" 수준

### Day 1~N: 매 git push마다 채워감
- 변경된 파일 + 관련 파일을 AI가 분석
- 테마 순회하며 해당 .md 생성/갱신
- 변경이 많은 파일일수록 문서가 빠르게 충실해짐

### Day 30+: 대부분 파일이 한 번 이상 변경됨
- 문서가 자연스럽게 채워진 상태
- 자주 변경되지 않는 파일: 선택적으로 "수동 전체 스캔" 옵션 제공

## 7. 비용/시간 최적화 방안

| 최적화 | 방법 | 효과 |
|--------|------|------|
| [판단] 단계 경량화 | Haiku 모델로 "필요한가? Yes/No"만 판단 | 판단 비용 ~90% 절감 |

| 배치 처리 | 중앙에서 스케줄 기반 일괄 빌드 | Docusaurus 빌드 횟수 최소화 |
| Full Reset | 매 테마마다 에이전트 신규 생성 (resume 금지) | 컨텍스트 누적 없이 1.0x 비용 유지 (Resume 시 1.8~3.6x) |

## 9. 참조 자료에서의 시사점

### SW-RCS-Documentation-구조분석.md에서:
- 사이드바 계층 구조를 페이지 단위 테마로 1:1 매핑하여 활용
- 실제 사이트 URL 기반으로 테마를 정의하여 불일치 방지
- 네트워크 포트, 기술 스택 등 정형 데이터는 규칙 기반 추출 가능

### SKILL_Standard_Report.md에서:
- **검증 게이트 패턴** (OMC Final_Checklist): 테마당 검증 단계의 설계 근거
- **순차적 실행 + 이전 결과 참조**: 테마 간 중복 방지를 위한 컨텍스트 누적 패턴
- **YAML frontmatter 표준화**: 메타데이터 기반 자동 분류의 검증된 접근
- **Good/Bad + WHY 패턴**: AI 프롬프트 설계 시 "이런 문서는 좋다/나쁘다 + 이유" 형태로 예시 제공하면 품질 향상
- **반복 제한 (Escalation)**: 검증 Fail 시 최대 2회 재시도 후 경고 태그 -- 무한 루프 방지
