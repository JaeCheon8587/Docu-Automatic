# Docusaurus 자동화 파이프라인 -- 상세 구현 계획 (v4)

## Context

`01.설계-결정사항.md`에서 확정된 4단계 사이클(`판단 -> md 작성 -> 검증 -> 저장`)을
Claude Code의 에이전트/스킬 체계로 구현하기 위한 상세 설계.

**핵심 설계 결정 (v4 확정):**
- **1단계 오케스트레이션**: Main CLI가 직접 테마 루프 + 판단 + 요구사항서 + Agent 위임 + 재시도 + 저장을 수행
- 에이전트 2개 (docu-writer + critic), 스킬 4개로 역할 분리
- 테마별 차이는 **Main이 작성하는 요구사항서의 내용**으로 구현
- **Full Reset 전략**: 매 테마마다 docu-writer/critic을 새로 생성 (이전 에이전트 resume 금지)
- 검증 실패 시 **Main 내부**에서 docu-writer 재호출 제어 (최대 2회)

**v3 -> v4 변경 이유**: Claude Code에서 서브 에이전트가 서브 에이전트를 생성할 수 없음 (Agent 중첩 제약). v3의 3단계 중첩(Main -> task-orchestrator -> docu-writer/critic)을 v4의 1단계(Main -> docu-writer/critic)로 평탄화.

**실행 환경**: Claude Code CLI
**실행 메커니즘**: Main CLI가 `skills/task-pipeline/SKILL.md`를 읽고, 테마마다 Agent 도구로 docu-writer/critic을 생성하여 위임

**가능성 근거 (기존 스킬 실측 비교):**
- plan --consensus: 순차 15회 호출 (Planner->Architect->Critic x 5) -> 안정 동작
- team: 수십 회 병렬+순차 호출 -> 안정 동작
- 우리 파이프라인: 순차 14~28회, 테마 간 독립, 병렬 없음 -> **기존보다 구조가 단순**

## 2. 테마별 실행 흐름

테마를 순차 순회. Main은 테마마다 판단+요구사항서를 작성하고, Agent(docu-writer/critic)를 호출하여 작성/검증을 위임한 후, 결과를 저장한다.

**1차 스코프 (4개 테마)**:
`intro -> getting-started/requirements -> architecture/overview -> architecture/component-diagram`

> 2차 확장: 제품별 페이지, 라이브러리, 개발 가이드 등

```
[Main CLI -- skills/task-pipeline/SKILL.md]
  테마 루프: theme in [intro, getting-started/requirements, architecture/overview, architecture/component-diagram]

  +-------------------------------------------------------------------+
  | [테마별 4단계 처리 -- Main이 직접 수행]                               |
  |                                                                   |
  |  Step A. [판단 + 요구사항서]                                        |
  |    (입력: git diff + theme-definitions의 해당 테마)                 |
  |    +-- git diff 또는 full-scan으로 코드 분석                        |
  |    +-- "이 변경이 {theme}에 필요한가?" 판단                          |
  |    +-- No -> SKIPPED 처리, execution-log 기록, 다음 테마             |
  |    +-- Yes -> 요구사항서 작성                                       |
  |           (관점, 대상 독자, 작성 스타일, 다뤄야 할 포인트)             |
  |                                                                   |
  |  Step B. [md 작성 -- Agent(docu-writer) 위임]                      |
  |    Agent 도구로 docu-writer 에이전트 생성 (NEW)                     |
  |    (입력: 요구사항서 + 관련 코드 + (재시도 시) critic 피드백)         |
  |    +-- .md 내용 반환                                               |
  |                                                                   |
  |  Step C. [검증 -- Agent(critic) 위임]                              |
  |    Agent 도구로 critic 에이전트 생성 (NEW)                          |
  |    (입력: md 내용 + theme_definition + 요구사항서)                   |
  |    +-- Pass/Fail + 피드백 반환                                      |
  |                                                                   |
  |  Step D. [재시도 판정 + 저장]                                       |
  |    +-- Pass -> .md 파일 저장 + execution-log PASS 기록              |
  |    +-- Fail (1~2회차) -> Step B로 복귀                              |
  |    |     (critic 피드백 + 원본 요구사항서, 새 에이전트 생성)            |
  |    +-- Fail (2회 초과) -> auto_generated_warning 태그               |
  |           + .md 파일 저장 + execution-log WARNING 기록              |
  +-------------------------------------------------------------------+

  +-------------------------------------------------------------------+
  | [중단 판정]                                                         |
  |  PASS/SKIPPED/WARNING -> consecutive_fail 초기화                    |
  |  FAIL -> consecutive_fail++                                        |
  |  consecutive_fail >= 3 -> 파이프라인 중단                            |
  +-------------------------------------------------------------------+
```

## 2.5 Full Reset 전략

**정책**: 매 테마 처리 시 docu-writer, critic을 모두 새로 생성한다. 이전 테마에서 생성된 에이전트를 재활용(resume)하지 않는다.

**근거:**

| 비교 항목 | Full Reset | Resume (재활용) |
|-----------|-----------|----------------|
| 컨텍스트 오염 | **없음** -- 매번 깨끗한 상태 | 있음 -- 이전 테마 잔여 컨텍스트가 판단 왜곡 가능 |
| 절차 준수 안정성 | **높음** -- 매번 스킬을 처음부터 읽음 | 낮음 -- 긴 컨텍스트에서 초기 지시 희석 가능 |
| 토큰 비용 (상대치) | **1.0x** (기준) | 1.8~3.6x (컨텍스트 누적으로 매 호출 토큰 증가) |
| 디버깅 용이성 | **높음** -- 테마별 독립 실행, 문제 격리 쉬움 | 낮음 -- 이전 테마와의 상호작용 추적 필요 |
| 구현 복잡도 | **낮음** -- 단순 생성/폐기 | 높음 -- 상태 관리, 컨텍스트 정리 로직 필요 |

**구현 방식:**
- Main이 `Agent` 도구로 docu-writer/critic을 호출할 때마다 새 서브 에이전트를 생성
- 재시도 시에도 docu-writer, critic을 새로 생성 (이전 에이전트 resume 금지)
- 이전 테마의 결과가 필요한 경우, 파일 경로를 전달하여 에이전트가 직접 읽도록 함 (컨텍스트 전달이 아닌 파일 시스템 경유)

## 3. 각 구성 요소 상세

### 3.1 task-pipeline (통합 오케스트레이션 스킬 -- Full Template)

**역할**: Main CLI의 전체 행동을 정의하는 유일한 오케스트레이션 스킬. **테마 루프 + 판단 + 요구사항서 + Agent 위임 + 재시도 + 저장**을 통합.

**실행 메커니즘**: Main CLI가 이 스킬을 읽고, 테마마다 판단/요구사항서를 직접 수행한 후 Agent 도구로 docu-writer/critic을 생성하여 작성/검증을 위임

**v4 변경 -- 역할 흡수:**
- v3의 task-pipeline 스킬(루프+저장)과 task-orchestrator 스킬(판단+요구사항서+재시도)을 통합
- Main이 직접 수행: 판단(Step A), 요구사항서 작성(Step A-5), 재시도 제어(Step D), 저장(Step D)
- Agent 위임: md 작성(Step B -> docu-writer), 검증(Step C -> critic)

**CRITICAL 제약 7개:**
1. .md 내용 직접 작성 금지 -> Agent(docu-writer)에 위임
2. 검증 직접 수행 금지 -> Agent(critic)에 위임
3. docu-writer 결과 수신 전 critic 호출 금지 (순차)
4. 테마 순회 반드시 순차 (병렬 금지)
5. 테마마다 새 docu-writer/critic 생성 (Full Reset)
6. 재시도 최대 2회 Hard Cap
7. 3연속 FAIL 시 파이프라인 중단

**안정성 장치:**
1. **execution-log.md** (파일 기반 상태 추적): 매 테마 완료 시 결과를 파일에 기록
2. **순차 대기 게이트**: Execution_Policy에 순차 제약을 CRITICAL로 명시
3. **반복 상한**: 3개 테마 연속 FAIL 시 파이프라인 중단

### 3.2 docu-writer (에이전트 + 스킬)

**역할**: Main의 요구사항서를 받아 .md 파일 내용 작성

**입력** (3개 입력 계약):
1. **requirement_spec** (항상 전달): theme, perspective, audience, writing_style, must_cover, do_not_cover, key_changes, related_files, output_path
2. **code_context** (항상 전달): 관련 소스 코드 파일 목록 및 내용
3. **critic_feedback** (재시도 시만 전달): critic이 지적한 문제 목록 (라인 번호 포함)

**출력**: YAML frontmatter 포함 .md 내용 (문자열)

**스킬 적용**: Lite Template
- 핵심 규칙: "요구사항서의 perspective와 writing_style을 엄격히 따를 것"
- "do_not_cover에 명시된 내용은 절대 포함하지 말 것"

### 3.3 critic (에이전트 + 스킬)

**역할**: docu-writer 결과물을 검증

**입력**:
- docu-writer가 작성한 md 내용
- theme-definitions/SKILL.md에서 현재 테마 블록
- Main이 작성한 요구사항서 (대조용)

**출력**:
```yaml
result: pass  # 또는 fail
frontmatter_valid: true
theme_fitness: pass  # 또는 fail
critic_feedback:
  - "tech-spec 영역의 구현 세부사항이 혼입됨 (라인 23-31)"
  - "sidebar_position 필드 누락"
```

**검증 기준 (2단계)**:
1. **frontmatter 검증** (규칙 기반, 7개 체크):
   - YAML frontmatter 블록(`--- ... ---`) 존재 여부
   - 9개 필수 필드 모두 존재 (title, sidebar_label, sidebar_position, section, theme, auto_generated, source_files, last_commit, generated_at)
   - `sidebar_position`이 숫자 타입
   - `theme`이 theme-definitions에 정의된 테마 ID 중 하나
   - `auto_generated`가 true
   - `source_files`가 비어있지 않음 (최소 1개)
   - `generated_at`이 ISO-8601 형식
2. **테마 적합성 검증** (AI 판단, 5개 관점):
   - perspective 일치: 문서 관점이 테마 정의와 부합하는가
   - do_not_cover 위반 스캔: 금지 내용이 혼입되지 않았는가 (라인 번호 기록)
   - must_cover 완전성: 필수 항목이 모두 다뤄졌는가
   - audience 적합성: 용어 수준과 설명 깊이가 대상 독자에 적합한가
   - writing_style 부합: 서술형/참조형 등 요구 스타일을 따르는가

**스킬 적용**: Lite Template

### 3.4 theme-definitions (참조 데이터)

**역할**: Main(task-pipeline 스킬)과 critic이 참조하는 페이지 단위 테마 정의

**내용**: 실제 Docusaurus 사이트 구조(http://110.110.10.70:8080/) 기반으로,
각 테마의 관점/대상 독자/작성 스타일/포함 범위/제외 범위를 구조화.
1차 스코프: intro, getting-started/requirements, architecture/overview, architecture/component-diagram (4개)

## 6. 검증 방법

1. task-pipeline/SKILL.md의 `<Steps>`가 Phase 1~3 + Step A~D를 모두 포함하는지 확인
2. **CRITICAL 제약 7개** 모두 `<Execution_Policy>`에 명시되는지 확인
3. **I/O contract 교차 검증**:
   - Main -> docu-writer 전달 = docu-writer Input_Contract (요구사항서 + 코드 + 피드백)
   - Main -> critic 전달 = critic Input_Contract (md 내용 + theme_definition + 요구사항서)
4. Agent 호출 리터럴 템플릿이 docu-writer.md, critic.md의 에이전트 정의와 일치하는지 확인
5. **Full Reset** 명시: task-pipeline/SKILL.md의 Execution_Policy에 CRITICAL 5로 기술
6. theme-definitions/SKILL.md의 테마가 실제 사이트 URL과 1:1 매핑되는지 확인
7. execution-log.md 기반 상태 추적이 task-pipeline/SKILL.md에 반영되는지 확인
8. 삭제된 파일(task-pipeline.md, task-orchestrator.md, task-orchestrator-skill/SKILL.md) 참조가 다른 파일에 남아있지 않은지 확인
