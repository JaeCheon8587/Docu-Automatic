# Docusaurus 자동화 파이프라인 — 상세 구현 계획 (v3)

## Context

`01.설계-결정사항.md`에서 확정된 4단계 사이클(`판단 → md 작성 → 검증 → 저장`)을
Claude Code의 에이전트/스킬 체계로 구현하기 위한 상세 설계.

**핵심 설계 결정 (v3 확정):**
- **2단계 오케스트레이션**: 메인(절차.md)은 루프 관리 + 상태 추적만, 작업 오케스트레이션이 테마별 판단/작성/검증 처리
- 에이전트 3개 (작업 오케스트레이션 + docu-writer + critic), 스킬 4개로 역할 분리
- 테마별 차이는 에이전트/스킬 개수가 아니라, **작업 오케스트레이션이 작성하는 요구사항서의 내용**으로 구현
- **Full Reset 전략**: 매 테마마다 모든 서브 에이전트를 새로 생성 (이전 컨텍스트 resume 금지)
- 검증 실패 시 **작업 오케스트레이션 내부**에서 docu-writer 재호출 제어 (최대 2회)

**실행 환경**: Claude Code CLI
**실행 메커니즘**: 메인 에이전트(Claude)가 절차.md를 스킬로 읽고, 테마마다 작업 오케스트레이션을 Agent 도구로 생성하여 위임

**가능성 근거 (기존 스킬 실측 비교):**
- plan --consensus: 순차 15회 호출 (Planner→Architect→Critic x 5) → 안정 동작
- team: 수십 회 병렬+순차 호출 → 안정 동작
- 우리 파이프라인: 순차 14~28회, 테마 간 독립, 병렬 없음 → **기존보다 구조가 단순**

## 1. 구성 요소 총괄

|-----------|------|------|------|

참조 파일: **1개** (테마 정의)

## 2. 테마별 실행 흐름

테마를 순차 순회. 메인은 테마마다 작업 오케스트레이션을 생성하여 위임하고, 반환된 결과를 저장한다.

**1차 스코프 (4개 테마)**:
`intro → getting-started/requirements → architecture/overview → architecture/component-diagram`

> 2차 확장: 제품별 페이지, 라이브러리, 개발 가이드 등

```
[메인 오케스트레이션 — 절차.md]
  테마 루프: theme in [intro, getting-started/requirements, architecture/overview, architecture/component-diagram]

  +-  메인 Step 1. [작업 오케스트레이션 생성] -------------------------+
  |  Agent 도구로 작업 오케스트레이션 서브 에이전트 생성                 |
  |  (전달: theme, section, output_path)                              |
  |  (자체 조달: git diff 실행, theme-definitions.md 읽기,            |
  |             관련 코드 파일 탐색 — 스킬이 지시)                      |
  |  ** Full Reset: 매 테마마다 새 에이전트 생성 (resume 금지) **       |
  +-------------------------------------------------------------------+

    +---------------------------------------------------------------+
    | [작업 오케스트레이션 내부 — 4단계 처리]                           |
    |                                                               |
    |  Step A. [판단 + 요구사항서]                                    |
    |    (입력: git diff + theme-definitions의 해당 테마)             |
    |    +-- "이 변경이 {theme}에 필요한가?" 판단                      |
    |    +-- No -> STATUS=SKIPPED 반환                               |
    |    +-- Yes -> 요구사항서 작성                                   |
    |           (관점, 대상 독자, 작성 스타일, 다뤄야 할 포인트)         |
    |                                                               |
    |  Step B. [md 작성]                                             |
    |    docu-writer 에이전트 생성                                    |
    |    (입력: 요구사항서 + 관련 코드)                                |
    |    +-- .md 내용 반환                                           |
    |                                                               |
    |  Step C. [검증]                                                |
    |    critic 에이전트 생성                                         |
    |    (입력: md 내용 + theme-definitions + 요구사항서)              |
    |    +-- Pass/Fail + 피드백 반환                                  |
    |                                                               |
    |  Step D. [재시도 판정]                                          |
    |    +-- Pass -> STATUS=PASS 반환                                |
    |    +-- Fail (1~2회차) -> Step B로 복귀                          |
    |    |     (critic 피드백 + 원본 요구사항서 전달)                    |
    |    +-- Fail (2회 초과) -> auto_generated_warning 태그           |
    |           + STATUS=WARNING 반환                                |
    +---------------------------------------------------------------+

  +-  메인 Step 2. [저장 + 상태 기록] --------------------------------+
  |  작업 오케스트레이션의 반환값을 파싱:                                |
  |    +-- SKIPPED -> 다음 테마로                                    |
  |    +-- PASS -> .md 파일 저장 + execution-log 기록                 |
  |    +-- WARNING -> .md 파일 저장 + 경고 로깅                       |
  |    +-- FAIL -> 에러 로깅 + 연속 Fail 카운트 증가                   |
  |  ** 작업 오케스트레이션 에이전트 폐기 (Full Reset) **               |
  +-------------------------------------------------------------------+
```

**작업 오케스트레이션 → 메인 반환 형식 (5필드):**

```yaml
STATUS: PASS | SKIPPED | WARNING | FAIL
THEME: getting-started/requirements
FILE: docs/getting-started/requirements.md
SUMMARY: "제품별 시스템 요구사항 테이블 신규 생성"
MD_CONTENT: |
  ---
  title: "시스템 요구사항"
  sidebar_label: "시스템 요구사항"
  sidebar_position: 1
  section: "getting-started"
  theme: "getting-started/requirements"
  auto_generated: true
  source_files:
    - "CMakeLists.txt"
    - "src/server/main.cpp"
  last_commit: "a1b2c3d"
  generated_at: "2026-03-05T14:30:00"
  ---
  # 시스템 요구사항
  ...
```

## 2.5 Full Reset 전략

**정책**: 매 테마 처리 시 작업 오케스트레이션, docu-writer, critic을 모두 새로 생성한다. 이전 테마에서 생성된 에이전트를 재활용(resume)하지 않는다.

**근거:**

| 비교 항목 | Full Reset | Resume (재활용) |
|-----------|-----------|----------------|
| 컨텍스트 오염 | **없음** — 매번 깨끗한 상태 | 있음 — 이전 테마 잔여 컨텍스트가 판단 왜곡 가능 |
| 절차 준수 안정성 | **높음** — 매번 스킬을 처음부터 읽음 | 낮음 — 긴 컨텍스트에서 초기 지시 희석 가능 |
| 토큰 비용 (상대치) | **1.0x** (기준) | 1.8~3.6x (컨텍스트 누적으로 매 호출 토큰 증가) |
| 디버깅 용이성 | **높음** — 테마별 독립 실행, 문제 격리 쉬움 | 낮음 — 이전 테마와의 상호작용 추적 필요 |
| 구현 복잡도 | **낮음** — 단순 생성/폐기 | 높음 — 상태 관리, 컨텍스트 정리 로직 필요 |

**구현 방식:**
- 메인(절차.md)이 `Agent` 도구로 작업 오케스트레이션을 호출하면, Agent 도구는 매번 새 서브 에이전트를 생성
- 작업 오케스트레이션 내부에서도 docu-writer, critic을 `Agent` 도구로 호출할 때마다 새로 생성
- 이전 테마의 결과가 필요한 경우, 메인이 파일 경로를 전달하여 작업 오케스트레이션이 직접 읽도록 함 (컨텍스트 전달이 아닌 파일 시스템 경유)

## 3. 각 구성 요소 상세

### 3.1 절차.md (메인 오케스트레이션 스킬 — Full Template)

**역할**: 메인 에이전트(Claude Code CLI)의 전체 행동을 정의하는 최상위 스킬. **테마 루프 관리, 작업 오케스트레이션 생성/폐기, 상태 추적**에 집중.

**실행 메커니즘**: Claude가 이 스킬을 읽고, 테마마다 Agent 도구로 작업 오케스트레이션을 생성하여 위임

**v3 변경 — 역할 축소:**
- v2에서 메인이 담당하던 판단 로직(테마별 필요 여부 판단 + 요구사항서 작성)을 작업 오케스트레이션으로 이관
- 메인은 루프 관리 + 작업 오케스트레이션 생성/폐기 + 반환값 파싱/저장 + 상태 추적에 집중

**SKILL_Standard_Report 기준 적용할 XML 태그:**
- `<Purpose>`: 각 제품의 git push → 테마 순회 → 작업 오케스트레이션 생성/위임 → .md 저장
- `<Use_When>`: git push CI 트리거 시
- `<Execution_Policy>`:
  - 테마 순회는 반드시 순차 (절대 병렬 금지)
  - CRITICAL — 작업 오케스트레이션의 반환을 받기 전에 다음 테마로 넘어가지 말 것
  - CRITICAL — 매 테마마다 새 작업 오케스트레이션을 생성할 것 (Full Reset)
  - 메인은 판단/요구사항서 작성/재시도 제어를 직접 수행하지 않음 (작업 오케스트레이션에 위임)
- `<Steps>`:
  - Phase 1: 입력 수집 (git diff, 관련 파일 경로)
  - Phase 2: 테마 루프 (테마마다 작업 오케스트레이션 생성 → 반환값 수신 → 저장/로깅 → 에이전트 폐기)
  - Phase 3: 완료 처리 (execution-log 최종 기록, docs-auto 브랜치 커밋)
- `<Escalation_And_Stop_Conditions>`:
  - 3개 테마 연속 FAIL → 전체 파이프라인 중단 + 로그
  - 작업 오케스트레이션 무응답 → 타임아웃 후 FAIL 처리 + 경고
- `<Final_Checklist>`:
  - [ ] 모든 테마 순회 완료
  - [ ] 경고 태그 문서 목록 로깅
  - [ ] docs-auto 브랜치에 저장 완료
  - [ ] execution-log.md 기록 완료

**안정성 장치 (기존 스킬에서 차용):**
1. **execution-log.md** (파일 기반 상태 추적, ralph의 prd.json 패턴):
   - 매 테마 완료 시 결과를 파일에 기록
   - 컨텍스트 손실 시에도 진행 상태 복구 가능
2. **순차 대기 게이트** (plan consensus의 CRITICAL 패턴):
   - Execution_Policy에 순차 제약을 CRITICAL로 명시
3. **반복 상한** (ralph 3-strike + plan 5회 상한 패턴):
   - 3개 테마 연속 FAIL 시 파이프라인 중단

### 3.2 작업 오케스트레이션 (에이전트 + 스킬)

**역할**: 단일 테마에 대한 4단계 사이클(판단 → md 작성 → 검증 → 재시도 판정)을 처리하는 중간 오케스트레이터

**입력**:

| 구분 | 항목 | 출처 |
|------|------|------|
| **메인에서 전달** | `theme` — 현재 처리할 테마 ID (예: `getting-started/requirements`) | 절차.md 루프 변수 |
| | `section` — 사이트 섹션 (예: `getting-started`) | theme_definition에서 추출 |
| | `output_path` — 저장될 .md 파일 경로 | 절차.md 계산 |
| **스킬 지시로 자체 조달** | git diff — 코드 변경 내역 | `git diff` 명령 실행 |
| | theme_definition — 해당 테마 정의 | `skills/theme-definitions/SKILL.md` 읽기 |
| | 관련 코드 파일 — 변경된 파일 + 연관 파일 | git diff 파싱 → 파일 읽기 |

**내부 흐름 (Step A~D):**

1. **Step A — 판단 + 요구사항서**: git diff와 테마 정의를 분석하여 문서화 필요 여부 판단. Yes이면 요구사항서 작성 (관점, 대상 독자, 작성 스타일, 포함/제외 범위)
2. **Step B — md 작성**: docu-writer 에이전트를 Agent 도구로 생성, 요구사항서 + 코드 전달, md 내용 수신
3. **Step C — 검증**: critic 에이전트를 Agent 도구로 생성, md 내용 + 테마 정의 + 요구사항서 전달, Pass/Fail 수신
4. **Step D — 재시도 판정**: Fail 시 docu-writer 재호출 (critic 피드백 포함, 최대 2회). 2회 초과 시 경고 태그 부착

**반환 형식 (5필드):**
```yaml
STATUS: PASS | SKIPPED | WARNING | FAIL
THEME: {theme_id}
FILE: {output_path}
SUMMARY: "변경 내용 한 줄 요약"
MD_CONTENT: |
  (YAML frontmatter 포함 .md 전체 내용)
```

**스킬 적용**: Full Template
- `<Purpose>`: 단일 테마의 4단계 사이클 처리 + docu-writer/critic 오케스트레이션
- `<Execution_Policy>`:
  - CRITICAL — docu-writer 결과를 받기 전에 critic을 호출하지 말 것
  - CRITICAL — critic 결과를 받기 전에 반환하지 말 것
  - CRITICAL — 재시도 시에도 docu-writer, critic을 새로 생성 (Full Reset)
  - 재시도 최대 2회 (Hard Cap)
- `<Escalation_And_Stop_Conditions>`:
  - docu-writer 무응답 → STATUS=FAIL 반환
  - 2회 초과 Fail → auto_generated_warning 태그 + STATUS=WARNING 반환

### 3.3 docu-writer (에이전트 + 스킬)

**역할**: 작업 오케스트레이션의 요구사항서를 받아 .md 파일 내용 작성

**입력** (3개 입력 계약):
1. **requirement_spec** (항상 전달): theme, perspective, audience, writing_style, must_cover, do_not_cover, key_changes, related_files, output_path
2. **code_context** (항상 전달): 관련 소스 코드 파일 목록 및 내용
3. **critic_feedback** (재시도 시만 전달): critic이 지적한 문제 목록 (라인 번호 포함)

**출력**: YAML frontmatter 포함 .md 내용 (문자열)

**스킬 적용**: Lite Template
- 핵심 규칙: "요구사항서의 perspective와 writing_style을 엄격히 따를 것"
- "do_not_cover에 명시된 내용은 절대 포함하지 말 것"

### 3.4 critic (에이전트 + 스킬)

**역할**: docu-writer 결과물을 검증

**입력**:
- docu-writer가 작성한 md 내용
- theme-definitions.md에서 현재 테마 섹션
- 작업 오케스트레이션의 요구사항서 (대조용)

**출력**:
```yaml
result: pass  # 또는 fail
frontmatter_valid: true
theme_fitness: pass  # 또는 fail
critic_feedback:
  - "tech-spec 영역의 구현 세부사항이 혼입됨 (라인 23-31)"
  - "sidebar_position 필드 누락"
```

**검증 기준 (2단계)**:
1. **frontmatter 검증** (규칙 기반, 7개 체크):
   - YAML frontmatter 블록(`--- ... ---`) 존재 여부
   - 9개 필수 필드 모두 존재 (title, sidebar_label, sidebar_position, section, theme, auto_generated, source_files, last_commit, generated_at)
   - `sidebar_position`이 숫자 타입
   - `theme`이 theme-definitions에 정의된 테마 ID 중 하나
   - `auto_generated`가 true
   - `source_files`가 비어있지 않음 (최소 1개)
   - `generated_at`이 ISO-8601 형식
2. **테마 적합성 검증** (AI 판단, 5개 관점):
   - perspective 일치: 문서 관점이 테마 정의와 부합하는가
   - do_not_cover 위반 스캔: 금지 내용이 혼입되지 않았는가 (라인 번호 기록)
   - must_cover 완전성: 필수 항목이 모두 다뤄졌는가
   - audience 적합성: 용어 수준과 설명 깊이가 대상 독자에 적합한가
   - writing_style 부합: 서술형/참조형 등 요구 스타일을 따르는가

**스킬 적용**: Lite Template
