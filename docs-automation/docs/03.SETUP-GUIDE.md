# 문서 자동화 시스템 셋업 가이드

## 개요

멀티 레포의 `docs/` 디렉토리를 자동 수집하여 단일 Docusaurus 사이트로 빌드/배포하는 시스템입니다.

### 아키텍처

```
소스 레포 (main branch push)
    → CI 트리거
    → AI .md 생성 (1단계 오케스트레이션)
    |   Main(task-pipeline 스킬): 4개 테마 루프
    |     → Agent(scout): 판단+요구사항서 → Agent(docu-writer) → Agent(critic) → 재시도/저장 + 로깅
    → docs-auto 브랜치에 .md Push
    → markdownlint + 보안 린트
    → 아티팩트 패키징 (tar.gz)
    → S3/MinIO 업로드
    → 중앙 빌드 트리거 (GitLab Pipeline Trigger API)

중앙 Docusaurus CI
    → 디바운스 (15초)
    → 모든 아티팩트 다운로드
    → 보안 검증
    → Docusaurus 빌드
    → rsync + 심볼릭 링크 원자적 배포
    → 헬스 체크 → 실패 시 자동 롤백
```

## Phase 1: 기본 인프라 (1~2주)

### 1.1 사전 요구사항

| 항목 | 요구사항 |
|------|---------|
| GitLab | 13.9+ (resource_group 지원) |
| Node.js | 18+ |
| Docker | 24+ |
| 배포 서버 | Nginx 1.25+, SSH 접근 |
| CI 러너 | 4코어, 8GB RAM, 50GB 디스크 |

### 1.2 중앙 Docusaurus 프로젝트 셋업

```bash
# 1. 레포 생성
cd central-docusaurus
npm install

# 2. 로컬 개발 서버 시작
npm start

# 3. 빌드 테스트
npm run build
```

### 1.3 아티팩트 저장소 (MinIO)

```bash
# Docker Compose로 MinIO 시작
docker-compose --profile storage up -d

# MinIO 콘솔: http://localhost:9001
# 초기 계정: minioadmin / minioadmin123 (반드시 변경)
```

### 1.4 소스 레포 설정

각 제품 레포에 아래 파일을 추가합니다:

**docs-manifest.json** (레포 루트):
```json
{
  "product_id": "{product-id}",
  "product_name": "{제품명}",
  "version": "1.0.0",
  "description": "VNC/RFB 기반 원격 데스크탑 제어 시스템",
  "docs_path": "docs/",
  "nav_label": "SW-RCS",
  "nav_position": 1,
  "allowed_extensions": [".md", ".mdx", ".png", ".jpg", ".svg", ".json"],
  "max_file_size_mb": 10
}
```

**.gitlab-ci.yml** (docs 관련 부분):
```yaml
include:
  - project: 'mirero/docs-central'
    file: '/ci-templates/source-repo-ci.yml'
    ref: main

variables:
  PRODUCT_ID: "{product-id}"
  CENTRAL_PROJECT_ID: "42"
  CENTRAL_TRIGGER_TOKEN: $DOCS_TRIGGER_TOKEN
```

### 1.5 배포 서버 준비

```bash
# 1. deployer 사용자 생성
sudo useradd -m -s /bin/bash deployer
sudo mkdir -p /var/www/docs/releases
sudo chown -R deployer:deployer /var/www/docs

# 2. SSH 키 배포
ssh-keygen -t ed25519 -C "docs-deploy" -f ~/.ssh/docs-deploy
# 공개키를 deployer@배포서버에 등록

# 3. Nginx 설정
sudo cp nginx/nginx.conf /etc/nginx/nginx.conf
sudo cp nginx/security-headers.conf /etc/nginx/conf.d/security-headers.conf
sudo nginx -t && sudo systemctl reload nginx
```

## Phase 1.5: AI 파이프라인 셋업

### 구성 요소

| # | 파일 | 유형 | 설명 |
|---|------|------|------|
| 1 | `skills/task-pipeline/SKILL.md` | 통합 오케스트레이션 스킬 | 4개 테마 루프 + Agent 위임(scout/docu-writer/critic) + 재시도 + 저장 |
| 2 | `skills/docu-writer-skill/SKILL.md` | docu-writer 스킬 | md 작성 규칙 |
| 3 | `skills/critic-skill/SKILL.md` | critic 스킬 | 검증 규칙 |
| 4 | `skills/theme-definitions/SKILL.md` | 참조 데이터 | 4개 테마 정의 |
| 5 | `skills/task-pipeline/SKILL-REFERENCE.md` | 참조 문서 | Examples/Failure_Modes/Tool_Usage |
| 6 | `agents/scout.md` | 에이전트 정의 | 탐색+판단+요구사항서 (Level 1 리프) |
| 7 | `agents/docu-writer.md` | 에이전트 정의 | md 작성기 (Level 1 리프) |
| 8 | `agents/critic.md` | 에이전트 정의 | 검증기 (Level 1 리프) |

### 실행 방법

```bash
# Claude Code CLI로 단일 제품 단일 테마 테스트
claude "절차.md 스킬을 실행해. 대상: {product-id}, 테마: overview만"
```

### 필요 환경변수

| 변수 | 설명 |
|------|------|
| `TARGET_REPO_PATH` | 대상 제품 레포 경로 |
| `DOCS_AUTO_BRANCH` | docs-auto 브랜치명 (기본: `docs-auto`) |

### 단일 테마 검증

```bash
# 1. 특정 레포에서 git diff 추출
cd /path/to/{product-id}
git diff HEAD~1 > /tmp/test-diff.txt

# 2. Claude Code로 단일 테마 실행
claude "절차.md 스킬 실행. git diff: /tmp/test-diff.txt, 테마: overview만, 저장: /tmp/test-output/"

# 3. 결과 확인
cat /tmp/test-output/overview.md
```

> 상세 아키텍처: `02.아키텍처.md` 참조.
> Full Reset 전략에 따라 매 테마마다 모든 에이전트를 새로 생성합니다.

## Phase 2: CI 파이프라인 자동화 (3~4주)

### 2.1 GitLab CI 변수 설정

중앙 프로젝트 → Settings → CI/CD → Variables:

| 변수 | 유형 | 보호 | 마스킹 | 설명 |
|------|------|------|--------|------|
| `S3_ACCESS_KEY` | Variable | Yes | Yes | MinIO 접근 키 |
| `S3_SECRET_KEY` | Variable | Yes | Yes | MinIO 시크릿 키 |
| `DEPLOY_SSH_KEY` | File | Yes | No | 배포 SSH 개인키 |
| `DOCKER_REGISTRY_USER` | Variable | Yes | Yes | 레지스트리 사용자 |
| `DOCKER_REGISTRY_PASS` | Variable | Yes | Yes | 레지스트리 비밀번호 |
| `SLACK_WEBHOOK_URL` | Variable | No | Yes | Slack 알림 웹훅 |

소스 레포별 → Settings → CI/CD → Variables:

| 변수 | 유형 | 보호 | 마스킹 | 설명 |
|------|------|------|--------|------|
| `DOCS_TRIGGER_TOKEN` | Variable | Yes | Yes | 중앙 빌드 트리거 토큰 (레포별 분리) |

### 2.2 파이프라인 트리거 토큰 생성

```bash
# 중앙 프로젝트에서 트리거 토큰 생성
# GitLab → Settings → CI/CD → Pipeline triggers → Add trigger
# 각 소스 레포에 개별 토큰 발급 (90일 주기 로테이션)
```

### 2.3 스케줄 빌드 (안전망)

```
GitLab → CI/CD → Schedules → New schedule
- Description: "Safety net build"
- Interval pattern: */30 * * * *
- Target branch: main
- Active: Yes
```

### 2.4 resource_group 확인

중앙 CI의 `.gitlab-ci.yml`에 `resource_group: docs-central-deploy`가 설정되어 있어 동시 빌드 경합을 방지합니다.

## 새 제품 추가 방법

### Step 1: 소스 레포에 docs-manifest.json 추가

```json
{
  "product_id": "new-product",
  "product_name": "New Product",
  "version": "1.0.0",
  "docs_path": "docs/",
  "nav_label": "New Product",
  "nav_position": 4
}
```

### Step 2: 소스 레포 CI에 문서 파이프라인 추가

`.gitlab-ci.yml`에 `source-repo-ci.yml` include.

### Step 3: 중앙 프로젝트에 docs 인스턴스 추가

`docusaurus.config.js`에 새 플러그인 인스턴스 추가:

```js
[
  '@docusaurus/plugin-content-docs',
  {
    id: 'new-product',
    path: 'product-docs/new-product',
    routeBasePath: 'new-product',
    sidebarPath: './sidebars/new-product.js',
  },
],
```

### Step 4: PRODUCT_IDS 변수 업데이트

중앙 CI의 `PRODUCT_IDS`에 새 제품 ID 추가:
```yaml
PRODUCT_IDS: "product-a product-b product-c new-product"
```

### Step 5: 네비게이션 업데이트

`docusaurus.config.js`의 `navbar.items`에 새 링크 추가.

## 문제 해결

### 빌드 실패

```bash
# 1. 로그 확인
# GitLab → CI/CD → Pipelines → 실패한 파이프라인 → 로그

# 2. 로컬에서 재현
cd central-docusaurus
npm run build 2>&1 | tee build.log

# 3. 아티팩트 수동 확인
mc ls minio/mirero-docs-artifacts/{product-id}/
```

### 롤백

```bash
# 자동 롤백: 헬스 체크 실패 시 CI에서 자동 실행

# 수동 롤백
./scripts/rollback.sh                    # 직전 릴리스로
./scripts/rollback.sh --release 20260301_120000  # 특정 릴리스로
```

### 사이드바 안 나옴

```bash
# 사이드바 수동 재생성
node scripts/generate-sidebars.js

# 또는 제품 레포에 커스텀 sidebars.js 배치
# product-docs/<product-id>/sidebars.js
```
